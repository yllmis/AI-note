## 一、MySQL MVCC 深度

### 三个隐藏列

| 隐藏列 | 说明 |
|--------|------|
| `DB_TRX_ID` | 最后修改该行的事务 ID |
| `DB_ROLL_PTR` | 回滚指针，指向 undo log 中上一个版本 |
| `DB_ROW_ID` | 隐含自增 ID（无主键时使用） |

### Undo Log 版本链

每行数据修改后，旧版本通过 `DB_ROLL_PTR` 串成链表，最新的在最前面。

```
当前行 → 版本1(TRX_ID=100) → 版本2(TRX_ID=80) → 版本3(TRX_ID=50)
```

### ReadView 四个字段

- `m_ids`：生成 ReadView 时**活跃未提交**的事务 ID 列表
- `min_trx_id`：m_ids 中的最小值
- `max_trx_id`：系统下一个要分配的事务 ID
- `creator_trx_id`：创建该 ReadView 的事务 ID

### 可见性判断（核心）

拿行的 `DB_TRX_ID` 去和 ReadView 比较：
1. `DB_TRX_ID == creator_trx_id` → 可见（自己改的）
2. `DB_TRX_ID < min_trx_id` → 可见（事务已提交）
3. `DB_TRX_ID >= max_trx_id` → 不可见（事务在 ReadView 之后开启）
4. `DB_TRX_ID` 在 `m_ids` 中 → 不可见（事务未提交）
5. 不在 m_ids 中 → 可见

如果不可见，沿 undo log 版本链向上找，直到找到可见的版本。

### RC vs RR 的区别

- **RC（Read Committed）**：每次 SELECT 都生成新的 ReadView
- **RR（Repeatable Read）**：只在第一次 SELECT 生成 ReadView，后续复用

## 二、Redis 在线状态最佳实践

### 面试中不能只说 key-value

**推荐方案**：`SET user:{uid}:online 1 EX 300`（5 分钟过期）

**心跳续期**：客户端每隔 30 秒发心跳，服务端续期 TTL

```go
// 心跳续期
func Heartbeat(uid string) {
    redis.Set(ctx, "user:"+uid+":online", 1, 5*time.Minute)
}
```

### Redis 宕机降级方案

1. **本地缓存**：用 `sync.Map` 或 `bigcache` 缓存在线状态，Redis 恢复后同步
2. **数据库兜底**：在线状态丢失后，下次消息投递时查数据库
3. **广播重建**：Redis 恢复后，客户端重新上报在线状态

### 常见追问

- **过期策略**：惰性删除 + 定期删除
- **内存淘汰**：LRU、LFU、allkeys-lru（推荐）
- **集群方案**：Redis Cluster，16384 个 slot 分片

## 三、Kafka 深度

### 顺序写原理

```
Partition 目录
├── segment-00000000000000000000.log    ← 数据文件
├── segment-00000000000000000000.index  ← 偏移量索引
├── segment-00000000000000100000.log
├── segment-00000000000000100000.index
```

- 每个 Segment 默认 1GB，满后切换新 Segment
- 写入时追加到当前 Segment 末尾，**不随机写**
- 消费时通过 `.index` 文件快速定位到偏移量

### 分区再平衡（Rebalance）

触发条件：
- 新增/删除消费者
- 消费者心跳超时
- 消费者处理消息超时

流程：
1. 协调者（Coordinator）检测到变化
2. 消费者组暂停消费
3. 重新分配分区（Range 或 RoundRobin 策略）
4. 消费者从上次提交的 offset 继续消费

### 手动提交 vs 自动提交

```go
// 自动提交（有丢失风险）
consumer.AutoCommit(true)

// 手动提交（推荐）
consumer.CommitOffsets() // 处理完业务后再提交
```

自动提交问题：消息还没处理完就提交了，消费者挂了 → 消息丢失。

### 高吞吐原因（完整版）

1. **顺序写磁盘**：追加写，不随机寻址
2. **Page Cache**：操作系统页缓存，减少磁盘 IO
3. **零拷贝**：sendfile 系统调用，数据直接从磁盘到网卡
4. **分区并行**：多 Partition 并行读写
5. **批量发送**：Producer 批量打包消息
6. **压缩**：消息压缩（Snappy/Gzip/LZ4）

## 四、面试回答技巧

### 回答模板

**技术问题**：
> 这个知识点我理解是 [核心概念]。具体来说 [展开 2-3 点]。在项目中 [结合实际场景]。

**项目问题**：
> 这个功能的核心目标是 [解决什么问题]。我采用了 [技术方案]，因为 [为什么选它]。遇到的问题是 [具体挑战]，通过 [怎么解决的]。

**不会的问题**：
> 这个点我不太确定，但我理解的大方向是 [说你知道的]。具体细节我需要再查一下。

### 项目包装公式

每个技术选型都要能回答：
1. **为什么选** — 解决什么问题，对比其他方案
2. **怎么用** — 具体实现细节
3. **遇到什么问题** — 性能、bug、设计缺陷
4. **怎么解决的** — 优化方案和效果（最好有数据）

### 常见追问准备

- "为什么不用 XX 而用 YY？"
- "如果流量增大 10 倍怎么办？"
- "数据量有多大？QPS 多少？"
- "有没有做过压测？"
