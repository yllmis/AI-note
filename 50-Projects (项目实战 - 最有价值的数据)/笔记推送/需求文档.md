# 笔记推送系统

根据用户当日学习任务及进度，定时推送相关学习资料至用户社交平台，供用户持续学习。

## 整体架构

```mermaid
graph LR
    A[git commit] --> B[post-commit hook 缓存 diff]
    B --> C[每日定时聚合当日 diffs]
    C --> D[DeepSeek V4 提炼知识点]
    D --> E[按文件夹层级归大类]
    E --> F[拼搜索词 + 调搜索 API]
    F --> G[去重过滤]
    G --> H[DeepSeek V4 生成摘要]
    H --> I[推送至飞书/邮件]
```

## 功能清单

1. 获取用户学习任务
2. 提炼用户学习任务的主要知识点
3. 根据知识点获取网上学习相关资料（或由 LLM 自行总结压缩）
4. 推送至用户邮箱或飞书等社交软件方便收取

## 产品形态：独立可运行程序（非 Obsidian 插件）

编译为单个 Go 二进制文件，通过 git hook + cron 运行，不依赖 Obsidian 进程。

| | Obsidian 插件 | 独立程序 ✓ |
|--|--|--|
| 触发方式 | 监听 vault 文件变更 | git hook 触发，符合需求设计 |
| 开发语言 | TypeScript，需学插件 API | Go，主力语言 |
| 后台能力 | 无，Obsidian 关了就不跑 | 独立进程，可跑在服务器/cron |
| LLM/搜索调用 | 受 Obsidian HTTP 限制 | 原生 net/http，无限制 |
| 部署 | 用户手动安装插件 | `go build` 出单二进制 |
| 适用范围 | 仅 Obsidian 用户 | 任何有 git 仓库的人 |

> [!tip] 为什么不做插件
> 触发机制是 git commit 而非 Obsidian 事件；推送需要后台定时运行，Obsidian 关闭后插件无法执行；Go 是主力语言，TypeScript 插件开发是额外成本。

## 技术栈

| 组件 | 选型 | 理由 |
|------|------|------|
| 主语言 | Go | 主力语言，飞书/邮件 SDK 成熟，编译单二进制 |
| 产品形态 | 独立 CLI 程序 | git hook + cron 驱动，不依赖 Obsidian 进程 |
| LLM | DeepSeek V4 | 性价比高，摘要/提炼质量足够 |
| 搜索源 | 掘金 API + Google Custom Search | 先双源，覆盖技术博客 + 通用网页 |
| 持久化 | SQLite | 轻量，存储已推送记录和 diff 缓存 |
| 推送 | 飞书 Bot Webhook + SMTP 邮件 | 官方 API 稳定，先行支持 |

## 功能分析

### 1. 学习任务获取

基于 git 提交记录驱动，commit = 一次学习行为。

**Vault 目录发现**：程序运行时默认取 `os.Getwd()` 作为 vault 路径（post-commit hook 从 vault 目录触发，天然一致）；后续支持 `config.yaml` 覆盖。

**Git 提交捕获（post-commit hook）**：

每次 commit 后自动触发，仅捕获 `.md` 文件变更：

```bash
#!/bin/bash
# .git/hooks/post-commit
git diff HEAD~1 HEAD -- '*.md' | 笔记推送 cache-diff --commit-hash=$(git rev-parse HEAD)
```

hook 脚本接收 diff 内容，写入 SQLite `pending_diffs` 表（字段：commit_hash / diff_content / created_at）。程序提供 `init` 命令自动安装 hook：`笔记推送 init` 将上述脚本写入 `.git/hooks/post-commit`。

**定时聚合**：每日定时任务（`笔记推送 push --daily`）从 `pending_diffs` 聚合当日所有待处理 diffs，送 LLM 提炼。处理完成后清理记录。

**兜底补丁**：若 hook 未安装（新 clone 的仓库等），聚合时额外从 `git log` 拉取当日提交：

```go
// 检查 pending_diffs 是否为空 + git log 有今日提交
// 对未缓存的 commit_hash 用 git show 补拉 diff
```

双重保障，hook 成功率接近 100%，兜底只是防御性设计。

> [!tip] 为什么用 git 而非文件 mtime
> commit 是有意动作，天然代表"学完了这批内容"；mtime 无法区分编辑和学习，改个逗号也会触发。

### 2. 知识点提炼

将当日 diffs 送入 DeepSeek V4，输出结构化知识点。

**输入**：当日聚合的 diff 文本

**Prompt 要点**：
- 从 diff 中提取所有新增/修改的知识点
- 每个知识点标注所属大类（复用 vault 文件夹层级：20-Go-Ecosystem / 30-Backend-Infra / 40-Architecture 等）
- 识别可延伸拓展的方向，每个大类列举 1-2 个延伸主题

**输出格式**（JSON）：

```json
{
  "topics": [
    {
      "category": "30-Backend-Infra",
      "keywords": ["Redis雪崩", "Redis击穿", "布隆过滤器"],
      "extensions": ["Redis集群方案", "热点Key探测"]
    },
    {
      "category": "20-Go-Ecosystem",
      "keywords": ["GMP调度", "Channel", "三色标记"],
      "extensions": ["goroutine泄漏排查", "GC调优"]
    }
  ]
}
```

> [!important] 大类定义
> 直接复用 vault 已有的**文件夹层级**作为大类，不需要 AI 额外分类。vault 的目录结构本身就是精心设计的分类体系。

### 3. 资料获取

按大类拼接搜索词，调搜索 API 拉候选文章。

**搜索词生成规则**：
- 主词：keywords 前 2 个 + category 中文映射（如"后端中间件 Redis雪崩 击穿"）
- 延伸词：extensions 中的主题各搜 1 篇

**搜索源优先级**：
1. 掘金 API — 技术文章密度高，中文为主
2. Google Custom Search — 兜底，覆盖博客/文档/论文

**质量过滤**：
- 排除已推送过的文章（查 SQLite `pushed_articles` 表，按 URL 去重）
- 每个大类最多返回 3 篇，延伸主题各 1 篇
- 若搜索结果 < 2 篇有效文章，由 DeepSeek V4 自行总结该知识点作为补充

**去重策略**：
- URL 精确匹配去重
- 标题相似度去重（简单 Jaccard，阈值 0.7）

### 4. 推送

将筛选后的文章推送至用户社交平台。

**推送时机**：每日定时聚合推送（建议每晚 20:00），而非每次 commit 即推。commit 粒度可能很碎，聚合后体验更好。

**推送格式**：

```
📚 今日学习推送 (2026-05-15)

━━ 30-Backend-Infra ━━
▸ Redis 雪崩与击穿实战方案
  https://juejin.cn/xxx
  摘要：从互斥锁到布隆过滤器，三种缓存防护策略对比...

▸ Kafka 分区副本机制详解
  https://blog.xxx/yyy
  摘要：ISR 副本同步流程与 Leader 选举原理解析...

━━ 20-Go-Ecosystem ━━
▸ GMP 调度模型深入
  https://xxx
  摘要：...

📎 延伸阅读
▸ Redis 集群方案对比 (延伸自缓存主题)
  https://xxx
```

**平台支持**：
- **飞书**：Bot Webhook，Markdown 卡片消息
- **邮件**：SMTP，HTML 格式

> [!note] 后续可扩展
> 微信（企业微信 Bot）、QQ（QQ Bot API）留后期，协议变动频繁。

## 数据存储设计

### SQLite 表结构

```sql
-- 待处理 diff 缓存
CREATE TABLE pending_diffs (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    commit_hash TEXT NOT NULL,
    diff_content TEXT NOT NULL,
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP
);

-- 已推送文章记录（去重用）
CREATE TABLE pushed_articles (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    url TEXT UNIQUE NOT NULL,
    title TEXT NOT NULL,
    category TEXT NOT NULL,
    pushed_at DATETIME DEFAULT CURRENT_TIMESTAMP
);

-- 每日推送日志
CREATE TABLE push_logs (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    push_date DATE NOT NULL,
    topic_summary TEXT,
    article_count INTEGER,
    status TEXT DEFAULT 'success',
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP
);
```

Vault 中已有的资源，避免重复造轮子：

| 资源 | 位置 | 复用方式 |
|------|------|----------|
| Smart Connections embeddings | `.smart-env/` | 候选文章的相似度重排序（后续增强） |
| Dataview 插件 | `.obsidian/plugins/dataview/` | 备选：按 tag/mtime 查询笔记 |
| frontmatter tags | 各笔记 YAML 头 | 搜索关键词的补充信号 |
| vault 文件夹结构 | 00~80 目录 | 大类分类体系 |
